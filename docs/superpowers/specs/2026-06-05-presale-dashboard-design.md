# Дашборд пресейла ОГВ — дизайн

**Дата:** 2026-06-05
**Владелец процесса:** Анна Кручинина
**Постановщик:** Никита Коротаев
**Источники требований:** транскрипция интервью 05.06.2026, файлы `Пресейл ОГВ. Статистика по выполненным работам(v14).xlsx`, `Пресейл ОГВ. Значения показателей выполненных работ(v1).xlsx`

## 1. Задача

Заменить ручной труд: сейчас Анна из большой таблицы «Статистика по выполненным работам» вручную, через сводные таблицы и фильтры, переносит показатели в маленький файл «Значения показателей выполненных работ». Нужен инструмент, который принимает большой файл и сам считает показатели, показывает их на наглядном дашборде с фильтрами и drill-down до конкретных запросов, а также умеет выгружать привычный маленький файл.

**Боли (из интервью):**
- Большой файл нечитаем для тех, кто не собирает статистику (Костя, Саша и др.).
- Ручной перенос данных через сводные — долго и подвержено ошибкам.
- Нет способа «провалиться» от цифры к списку конкретных запросов.

## 2. Решения по объёму (приняты с заказчиком)

- **Размещение:** отдельное приложение (не внутри MCP-Directum-RX), тот же стек.
- **Главный результат:** интерактивный дашборд + экспорт в Excel.
- **Поступление данных:** большой файл кумулятивный (январь–текущий месяц, все статусы). При каждой загрузке вся годовая матрица пересчитывается заново.
- **Доступ:** общий сервер. Анна загружает файл раз в месяц, остальные открывают URL и фильтруют. Сервер хранит последний загруженный датасет.
- **Drill-down:** обязателен (must-have).
- **Компоновка:** вариант A — KPI-карточки сверху, строка фильтров, вкладки по показателям (график + таблица), клик по ячейке → панель списка запросов.
- **Хранение:** SQLite.

## 3. Стек и ограничения

- Python 3.10+, FastAPI, Uvicorn.
- Frontend: Vanilla JS, plain HTML/CSS. **Без React/Vue/Angular.**
- Графики: Chart.js, хранится **локально** (без CDN — Docker без интернета).
- Excel: openpyxl (чтение и запись).
- Тесты: pytest + Playwright E2E. Покрытие ≥70%.
- TDD: тесты пишутся до реализации.
- Общение с пользователем — на русском.

## 4. Архитектура

```
presale-dashboard/
├── src/
│   ├── main.py            # FastAPI: эндпоинты
│   ├── parsing.py         # чтение листа «Исх данные» из xlsx → нормализованные записи
│   ├── metrics.py         # расчёт показателей и разрезов (чистые функции)
│   ├── export.py          # генерация «Значения показателей.xlsx»
│   ├── storage.py         # SQLite: запись/чтение датасета и истории загрузок
│   ├── config.py          # настройки (порт, путь к БД)
│   └── models.py          # pydantic-схемы
├── static/                # index.html, app.js, style.css, vendor/chart.min.js
├── tests/
│   ├── unit/              # parsing, metrics, export
│   ├── fixtures/          # парные xlsx для golden-теста
│   └── e2e/               # Playwright
└── data/                  # presale.db (gitignore)
```

**Поток данных:**
1. Анна загружает `*.xlsx` → `POST /api/upload`.
2. `parsing.py` читает лист «Исх данные», валидирует колонки, нормализует строки.
3. При успехе `storage.py` атомарно заменяет содержимое таблицы `status_events` и пишет запись в `uploads`.
4. Дашборд (`GET /api/metrics?...`) и экспорт (`GET /api/export?...`) считают показатели из SQLite с учётом фильтров.
5. Drill-down (`GET /api/requests?...`) возвращает список запросов за конкретной ячейкой.

## 5. Модель данных (SQLite)

**Таблица `status_events`** — одна строка листа «Исх данные» = одна запись:

| Поле | Источник (колонка) | Тип |
|---|---|---|
| id_запроса | ИД запроса | TEXT |
| запрос | Запрос на пресейл | TEXT |
| организация | Организация | TEXT |
| продукт | Продукт | TEXT |
| масштаб | Масштаб | TEXT |
| услуга | Услуга | TEXT |
| инициатор | Инициатор | TEXT |
| команда | Команда | TEXT |
| бизнес_единица | Бизнес-единица | TEXT |
| статус | Статус | TEXT |
| предыдущий_статус | Предыдущий статус | TEXT |
| дата_начала_статуса | Дата начала статуса (в файле — строка) | TEXT |
| дата_окончания_статуса | Дата окончания статуса (строка) | TEXT |
| месяц | **колонка «Месяц начала текущего статуса»** (готовая, 1–12) | INTEGER |
| длительность_рд | Длительность, р.д. | REAL |
| длительность_проработки_рд | Длительность проработки запроса (раб. дн.) | REAL |
| отработано_часов | Отработано часов | REAL |
| примечание | Примечание | TEXT |

**Таблица `uploads`** — история: `id, имя_файла, дата_загрузки, кол_во_строк, загрузил`.

## 6. Правила расчёта показателей

Группировка по **колонке «Месяц начала текущего статуса»** строки соответствующего статуса; дедуп по `id_запроса`.

> **Валидация (выполнена на этапе дизайна):** расчёт по этим правилам на `Статистика (v14)` воспроизводит `Значения показателей (v1)`: **Поступило** и **Проработано** — счётчики совпадают 1-в-1 по всем услугам и месяцам; **Ср. трудоёмкость** совпадает везде, кроме одного апрельского запроса, переклассифицированного между снимками (услуга сменилась «Оценка сопровождения»↔«Оценка обновления»). Это ожидаемый дрейф свежих месяцев, не ошибка логики.

| Показатель | Что считаем | Фильтр статуса | Месяц по | Дедуп |
|---|---|---|---|---|
| **Поступило** | кол-во запросов | `Статус = Инициализация` | строке «Инициализация» | distinct id_запроса |
| **Проработано** | кол-во запросов | `Статус = Принято` | строке «Принято» | distinct id_запроса |
| **Ср. трудоёмкость** (ч) | среднее `Отработано часов` (одно значение на запрос) | запросы со статусом `Принято` | строке «Принято» | distinct id_запроса |
| **Ср. длительность** (раб. дн) | на запрос: сумма `Длительность проработки запроса (раб. дн.)` по строкам запроса (статусы продавца там уже исключены), затем среднее по запросам | запросы со статусом `Принято` | строке «Принято» | distinct id_запроса |
| **Ср. на контроле** (раб. дн) | на запрос: сумма `Длительность, р.д.` по строкам со статусом `На контроле`, затем среднее | запросы со статусом `Принято` | строке «Принято» | distinct id_запроса |

**Дедуп перенаправленных:** при перенаправлении из соседней БЕ запрос может иметь две строки «Инициализация» (count=2). Считаем такой запрос как 1 (distinct по id_запроса).

**Разрезы (выбираемое измерение строк):** Услуга / Продукт / Масштаб / Инициатор / Команда. Для трудоёмкости и длительности приоритетны Услуга и Команда.

**Принадлежность месяцу для трудоёмкости/длительности/на-контроле:** запрос относится к месяцу, в котором у него начался статус «Принято».

## 7. Дашборд (вариант A)

- **Шапка:** название · дата последней загрузки · кто загрузил · кнопки «⬆ Загрузить файл», «⬇ Выгрузить Excel».
- **KPI-карточки:** Поступило, Проработано, Ср. трудоёмкость, Ср. длительность, Ср. на контроле — итоги за выбранный период.
- **Строка фильтров:** Месяц (мультивыбор), Услуга, Продукт, Масштаб, Команда, Инициатор. Применяются ко всему дашборду.
- **Вкладки по показателям.** На каждой:
  - переключатель «Разрез по: Услуга / Продукт / Масштаб / Инициатор / Команда»;
  - график (столбики по месяцам для счётных показателей, линии для средних) + таблица «месяц × разрез» рядом;
  - **клик по ячейке → панель drill-down:** список запросов за этим числом — название, организация, услуга, команда, отработано часов, статус, даты. Из панели — экспорт списка.

## 8. Экспорт в Excel

Кнопка генерирует файл в формате текущего «Значения показателей выполненных работ»: листы **Поступило / Проработано / Ср. трудоёмкость / Ср. длительность** (+ **Ср. на контроле**). Строки = выбранный разрез, столбцы = месяцы (январь–декабрь). Учитывает активные фильтры.

## 9. Ошибки и валидация

- Проверка наличия листа «Исх данные» и обязательных колонок при загрузке; понятное русскоязычное сообщение об ошибке формата.
- Загрузка атомарна: новый датасет заменяет старый только при успешном парсинге всего файла.
- Пустые/отсутствующие значения в средних показателях не учитываются (как в Excel: пусто ≠ 0).

## 10. Тестирование

- **Unit:** `parsing` (нормализация, валидация колонок), `metrics` (каждый показатель + дедуп перенаправленных + все разрезы + принадлежность месяцу), `export` (структура листов).
- **Golden-тест:** прогон расчёта на приложенном `Статистика по выполненным работам (v14)` и сравнение результата с `Значения показателей (v1)` как с эталоном. Учесть, что значения свежих месяцев в v14 могли измениться относительно снимка v1 (доотмеченные часы, смена статусов) — логика расчёта и ранние месяцы должны совпадать, поздние сверяются с допуском.
- **E2E (Playwright):** загрузка файла → применение фильтра → drill-down по ячейке → выгрузка Excel. Скриншоты → `tmp/screenshots/`.
- **Покрытие ≥70%.**

## 11. Зависимости и открытые вопросы

- Валидация выполняется по двум приложенным к постановке файлам: `Статистика по выполненным работам (v14)` (источник) и `Значения показателей (v1)` (эталон). Дополнительных файлов не требуется.
- Колонка `Длительность проработки запроса (раб. дн.)` присутствует в файле Анны (её формула, исключающая статусы продавца). Если в будущем сырой выгрузке из Ауры её нет — потребуется считать самим из `Длительность, р.д.` по статусам «Инициализация»+«В работе» (фолбэк).

## 12. Вне объёма (YAGNI)

- Авторизация/роли пользователей (внутренний инструмент, общий доступ).
- Накопление истории по месячным срезам (файл уже кумулятивный).
- Редактирование данных в приложении (источник истины — выгрузка из Ауры).
- Прогнозирование/ML (упоминалось как «может быть в будущем», но не сейчас).
